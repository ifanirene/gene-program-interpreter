48-Hour Claude Skill MVP Plan

Transform the existing gene-program annotation pipeline into a Claude-native Hackathon demo with parallel, auditable literature research—while preserving a clean path to a publication-quality product.

1. Demo promise

The user supplies weighted gene programs, experimental context, and optional Perturb-seq regulator effects. The Skill validates the inputs, launches one isolated literature agent per selected program, verifies the returned evidence, and renders an interactive HTML interpretation report.

MVP scope: demonstrate 3–5 representative programs, not the full dataset. Include one strong program, one ambiguous program, and one context/directionality counterfactual.

2. Minimal architecture

Existing deterministic preprocessing
        ↓
program_bundles/{program_id}.json
        ↓
Python asyncio orchestrator (3–5 concurrent Agent SDK sessions)
        ↓
Project Skill + PubMed MCP + OpenAlex MCP
        ↓
research_results/{program_id}.json
        ↓
Deterministic verifier and deduplicator
        ↓
Claude synthesis using verified evidence only
        ↓
Existing HTML report renderer

Key decision: control parallelism in Python rather than asking a manager agent to delegate all programs. Each program becomes independently retryable, cacheable, inspectable, and cost-limited.

4. Per-program literature agent contract

  1. Read exactly one program bundle and the shared research protocol.
  2. Search direct cell-type/tissue/condition evidence first; broaden only when direct evidence is sparse.
  3. Use PubMed for biomedical retrieval and OpenAlex for identifier resolution, related-work expansion, citation links, and metadata cross-checking.
  4. Investigate four questions: program function, regulator mechanism, perturbation-direction consistency, and competing or contradictory explanations.
  5. Return 1–3 candidate mechanisms and a small set of high-value papers; explicitly report evidence gaps.
  6. Reference only tool-returned evidence IDs. Never invent a PMID, DOI, quotation, or full-text conclusion.
  7. Do not assign the final program label. The downstream synthesis step compares verified evidence across programs.

Quality target: comparable to a strong interactive research agent for this bounded task—not a general-purpose deep-research system. The template, tools, context isolation, contradiction search, and verifier are what make the output reliable.

6. Two-day execution plan

Day 1 — Skill and research fan-out
  • Freeze the 3–5 demo programs and input semantics.
  • Create SKILL.md, the research protocol, and JSON schemas.
  • Generate one program bundle from the current preprocessing output.
  • Connect read-only PubMed and OpenAlex MCP tools.
  • Run one program end-to-end, then add bounded parallel execution.
Day 2 — Verification and demo polish
  • Add identifier, context, direction, and duplicate checks.
  • Integrate verified evidence with the existing annotation/report code.
  • Add cache, retry, timeout, and deterministic fallback behavior.
  • Pre-run and cache the demo programs.
  • Write a one-command demo and a short README walkthrough.

7. Five-minute demo flow

  1. Ask the Skill to interpret a selected aged-hepatocyte program.
  2. Show parallel program-level research status and the tools being used.
  3. Open one claim and trace it to genes, regulators, PubMed/OpenAlex records, and context matching.
  4. Remove regulator directionality or change the biological context and rerun the synthesis from cached evidence.
  5. Show the ambiguous program abstaining or presenting two competing hypotheses.

8. Reliability gates

  • Read-only MCP tools and explicit allowlists.
  • Concurrency semaphore, maximum agent turns, paper cap, and per-program cost cap.
  • Check MCP connection status before research begins.
  • Cache raw tool records separately from model summaries.
  • Missing or unresolved evidence becomes unsupported, not a warning hidden in prose.
  • If an agent fails, retain deterministic enrichment and label the literature section incomplete.

9. Explicitly defer

POST-HACKATHON Full evidence graph, calibrated confidence, broad database coverage, all-program batch runs, expert benchmark, generalized UI, user accounts, and publication claims. The MVP should expose interfaces for these additions without implementing them now.

10. Acceptance criteria

  • One command launches the Skill workflow and produces a complete HTML report.
  • All selected programs research concurrently without manual prompt editing.
  • Every displayed scientific claim links to one or more resolvable evidence records.
  • The report visibly separates supported, partial, contradictory, and missing evidence.
  • The demo can rerun from cache when an MCP service or network call fails.
Recommended implementation stance: reuse the current deterministic preprocessing and HTML renderer; replace only the manual literature stage and the free-form evidence interface during the Hackathon.
