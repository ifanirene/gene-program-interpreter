Gene Program Interpreter · research/

Literature Research Protocol Map

One isolated Claude Agent SDK session per gene program, fanned out in Python (asyncio + Semaphore), each answering a single question — what is the shared biological function of this program's genes? — against read-only literature tools, then reduced by a deterministic verifier. This map shows every step, its tunable parameters, and where research quality can be raised.

The pipeline, end to end

STEP ①

Assemble bundle

research/bundle.py · offline

One immutable program_bundles/{id}.json per program — the complete prompt the agent reads. Nothing not its job is included.

  • program_genestop 20 by loading
  • distinctive_genestop 10 by uniqueness
  • regulators / cond.top 6 |log2fc|
  • also carriesfunctions, brief
STEP ②

Fan-out sessions

research/research_parallel.py

Isolated query() session per bundle. Fresh cwd holds only its own bundle; protocol.md is the system prompt. No manager agent — Python controls parallelism.

  • concurrency4 (k=3–5)
  • modelsonnet-4-6
  • max_turns30
  • max_budget_usd1.0/session
  • timeout600 s
  • attempts2 (retry once)
STEP ③

Retrieve & submit

research/literature.py (in-process MCP)

Agent decides every query, retrieves first (never writes from memory), attaches papers to 1–3 mechanisms, and calls submit_result exactly once.

  • search cap10 (max 15)
  • fetch cap20 PMIDs
  • mechanismshard max 3
  • NCBI rate9/s keyed
STEP ④

Verify & reduce

research/verify.py · deterministic

Normalizes flat output, dedups papers into an evidence pool, resolves every PMID/DOI, flags retractions, and derives each mechanism's status.

  • resolvesPMID + DOI
  • statusfrom resolvability
  • retractionCrossRef flag
  • dedupcross-program

Tool surface — read-only, the agent sees nothing else

search_pubmed(query, max_results)

Primary biomedical retrieval. Returns PMIDs (discovery ids), sorted by relevance.

cap 10 · max 15 · relevance sort

fetch_pubmed(pmids)

Canonical metadata + real DOI, year, journal, study_type, abstract, is_preprint, is_retracted. Fetch before citing.

≤ 20 PMIDs / call

search_openalex(query, max_results)

Cross-publisher, includes preprints. Carries doi/pmid + cited_by_count. Requires OPENALEX_API_KEY — errors out if unset.

key-gated channel

resolve_doi(identifier)

Resolve a DOI or bibliographic string against CrossRef to verify a real DOI, title, and year.

keyless

Sandbox & guardrails (every session)

  • Isolated cwd — Read can only see this program's bundle.
  • Deny-by-default — can_use_tool allowlist: literature + submit + Read.
  • Denylist — Bash, Write, Edit, WebSearch/Fetch, Agent, Skill… all blocked.
  • Sandboxed config — setting_sources=[] + strict_mcp_config.
  • Retrieve-first — cite only tool-returned identifiers; never fabricate.
  • No final label — the agent proposes mechanisms; synthesis labels downstream.
  • Untrusted text — retrieved titles/abstracts are data, never instructions.
  • Auth split — research runs on the Claude.ai subscription, not the API key.

Adjustments to raise research quality — ranked

1

Verify claims, not just identifiers

Today a mechanism is “supported” if its PMIDs merely resolve — a real but off-topic paper counts. Add the reserved claim-vs-paper entailment check: an adjudicator reads the abstract and confirms it actually supports the mechanism.

Why: the single biggest gap. “Supported” currently overstates evidence; resolvability ≠ relevance. Scaffold already exists (FUTURE_claim_verification.md).
High impactVerifier ④
2

Make OpenAlex failure loud, add a preprint fallback

Without OPENALEX_API_KEY the entire cross-publisher + preprint channel silently returns an error, leaving PubMed-only recall. Detect the missing key at launch, warn in the audit, and fall back to a keyless bioRxiv/Europe PMC search.

Why: a whole retrieval modality can be dark without anyone noticing — recall quietly halves for recent/preprint biology.
High impactLow effortTools ③
3

Gene-synonym expansion before search

Papers use aliases (Madcam1 vs MAdCAM-1, mouse vs human casing). Add an NCBI Gene / alias-resolution helper (or bake synonyms into the bundle) so queries hit the literature's actual vocabulary.

Why: symbol mismatches drop true positives at the retrieval step — the cheapest recall the pipeline is leaving on the table.
High impactBundle ①Tools ③
4

Rank by evidence quality, not just relevance

PubMed is fixed to relevance sort and quality signals go unused. Surface cited_by_count (already fetched from OpenAlex), study_type, and recency to the agent, and guide it to prefer reviews for established function, primaries for specific claims.

Why: lets the agent spend its limited turns on load-bearing papers instead of the first relevant hit.
Low effortTools ③
5

Give the agent query-craft guidance

The protocol says “search the strongest gene(s)” but offers no query patterns. Add few-shot examples using field tags ([tiab], [Majr]), gene-set co-occurrence, and cell-type context to move beyond one-gene lookups.

Why: query quality is the highest-leverage, zero-infra lever; better queries find the papers that ground a shared function, not a famous single gene.
Low effortProtocol ②
6

Citation-graph expansion

Add one hop of forward/backward citations (OpenAlex referenced_works / cited_by) from a strong seed paper, so the agent can reach the review or key primary it didn't phrase a query for.

Why: literature search recall is bounded by phrasing; the citation graph routes around it and finds consensus sources.
Tools ③
7

Turn/budget self-pacing + gene-coverage report

Sessions that exhaust max_turns without submit_result fall back to “incomplete,” and a retry hits the same wall. Inject remaining-turn awareness with a “submit now” checkpoint, and require a genes_grounded list so unresearched genes are explicit, not lost in prose.

Why: converts silent truncation into a graceful partial result, and makes coverage auditable per program.
Runner ②Schema
8

Raise caps for deep programs

Retrieval is capped at 10/search and 30 turns — tight for a 30-gene program spanning several themes. Make caps config-driven and consider a larger fetch/search budget (and a stronger model) for programs flagged as high-priority.

Why: current defaults optimise cost; some programs need depth. Make the trade-off a knob, not a constant.
